## 3. 강점

### 3.1 Step 1 범위가 현실적임

Delta는 완성형 Knowledge Graph를 한 번에 구현하려 하지 않고, 현재 바로 효과가 있는 pre-filter와 RCA Markdown 저장부터 시작한다.

```text
log.txt → issue-type pre-filter → signal_<issue_type>.txt → Claude/Roo 분석 → RCA_<date>_<issue_type>.md 저장
```

이 흐름은 현재 사용 방식과 자연스럽게 연결되며, 실제 업무 도입 부담이 낮다.

### 3.2 500회 이상 반복 분석 관점이 반영됨

추가된 파일 기반 Knowledge Graph 운영안은 아래 구조를 제안한다.

```text
case YAML + signal txt + index YAML/Markdown + skill seed Markdown
```

이는 다음 장점을 가진다.

- Claude/Roo가 직접 읽을 수 있음
- Git diff로 변경 추적 가능
- 초기 운영 부담이 낮음
- 500건 이후 SQLite 또는 graph DB로 확장 가능
- 원본 대용량 로그와 구조화 지식을 분리 가능

### 3.3 원본 로그 저장 정책이 안전함

Delta는 원본 `log.txt`를 Knowledge Graph에 직접 저장하지 않고 path, size, checksum, time range만 저장하도록 제안한다. 이는 용량, 보안, LLM context 측면에서 타당하다.

### 3.4 Skill Seed 누적 구조가 명확함

문제 유형별 `skills_seed/*.md` 운영안은 현재 skill이 없는 상태에서 시작하기 좋은 방식이다. 초기 RCA Markdown과 case YAML을 축적한 뒤, 10~20건부터 공통 checklist와 trigger signature를 skill seed로 정제하는 흐름이 적절하다.

## 4. 보완 필요 사항

### 4.1 Prompt 반영 범위 결정 필요

Delta에는 Step 1 pre-filter뿐 아니라 500회 이상 운영을 위한 파일 기반 Knowledge Graph 구조도 포함되어 있다.

따라서 실제 `prompt/5_5_rca_knowledge_graph_py_package_prompt.md` 반영 시 아래 중 선택이 필요하다.

| 선택 | 내용 | 장점 | 단점 |
|------|------|------|------|
| A | Step 1 pre-filter와 RCA Markdown만 반영 | 작고 즉시 반영 가능 | 500회 운영안은 별도 문서에만 남음 |
| B | Step 1 + 파일 기반 KG 운영안 요약까지 반영 | 5.5 방향성이 명확해짐 | prompt가 다소 길어짐 |
| C | Step 1은 prompt에 반영, 500회 운영안은 별도 `rca_kg/README` 후보로 분리 | 구현/운영 분리 명확 | 추가 산출물 필요 |

**권장:** B. 5.5의 핵심이 Knowledge Graph이므로, prompt에는 Step 1뿐 아니라 장기 저장 구조의 요약까지 포함하는 것이 좋다. 다만 전체 case YAML 예시는 Delta에 유지하고 prompt에는 짧은 schema 예시만 넣는 방식이 적절하다.

### 4.2 `issue_type` naming 표준 필요

Delta에서 문제 유형 표기가 일부 혼재한다.

```text
rach failure / rach_failure
scgFailure / scg_failure
tx abnormal / tx_abnormal
L2 max retransmission / l2_retx
crash
```

실제 schema와 index에서는 snake_case를 표준으로 고정하는 것이 좋다.

권장 표준:

```text
rach_failure
scg_failure
tx_abnormal
l2_max_retransmission
crash
```

표시명은 사람이 보는 문서에서만 유지한다.

### 4.3 Windows 환경 기준 pre-filter 구현 표현 필요

초기 논의에는 `grep`/`awk` 예시가 있었지만, 현재 프로젝트 환경은 Windows 11 + PowerShell이다. 실제 prompt 반영 시 특정 shell script를 강제하지 않고 다음 원칙을 두는 것이 안전하다.

```text
- 구현자는 현재 환경을 확인한다.
- PowerShell, Python, Git Bash 중 사용 가능한 도구를 선택한다.
- 기본 구현은 Python 또는 PowerShell로도 가능하게 둔다.
```

### 4.4 Schema validation은 Step 1에서는 선택으로 두는 것이 적절

500회 이상 운영을 고려하면 schema validation이 중요하지만, Step 1에서 필수로 넣으면 착수 부담이 커진다.

권장:

```text
Step 1: RCA Markdown + case YAML template
Step 2: schema validation 추가
```
