v0.39는 신규 standalone 프롬프트를 `5.0.1`로 명명한다.

그러나 `5_0_common_automation_framework.md`에는 이미 `5.0.1`이 존재한다.

```text
## 5.0.1 Environment Discovery   ← 기존

...
```

즉 `5.0.1`은 5.0 프레임워크의 **하위 절(subsection) 시리즈 5.0.1~5.0.10**의 일부로 이미 점유된 번호다. 같은 번호 `5.0.1`이 한쪽은 "Environment Discovery"(5.0 내부 절), 다른 쪽은 "독립 프롬프트 파일"을 동시에 가리키게 된다.

- 리스크 1: 사용자/Roo/Claude가 "5.0.1을 수행해줘"라고 했을 때 어느 것을 의미하는지 모호하다.
- 리스크 2: `00_overview.md`의 구조 표(`5_0`, `5_1`, `5_2`...)에 `5_0_1` 파일을 추가하면, 파일 레벨 번호 체계와 5.0 내부 절 번호 체계가 충돌한다.
- 리스크 3: 향후 Master 본문에 이 항목을 반영할 때 5.0 절 번호와 정면 충돌한다.

**완화 방안 (택1, S5에서 결정 필요):**

```text
옵션 A (권장): 신규 프롬프트를 5.0.1이 아니라 별도 파일 번호로 부여
```

## [KEEP] 5.0.1을 5.3에서 분리하는 구조 (번호 문제와 별개로 분리 자체는 타당)

Discovery 방식을 5.3 내부에 두지 않고 공통 프롬프트로 분리하는 설계는 타당하다.

- 주간보고 Parent Page는 매주 새로 생성되므로 "특정 주차 데이터"는 재사용 가치가 낮다.
- 반면 "Child Page를 찾는 방식"은 재사용 가치가 높다.
- 이 분리는 "재사용 가능한 discovery method만 저장"이라는 명시된 의도와 정확히 일치한다.

→ 분리 구조는 KEEP. 단 명칭/번호만 위 [RISK] 항목대로 조정.

## [KEEP] profile이 주차별 데이터를 저장하지 않는 설계

profile에 Parent Page ID / URL / Child Page URL·ID 목록을 저장하지 않고, discovery method·confidence·fallback list만 저장하는 설계는 올바르다.

- Parent Page가 매주 새로 만들어진다는 운영 현실과 일치.
- profile이 "method 메모리"로만 기능하므로 주차 간 안전하게 재사용 가능.

→ KEEP. 설계 의도가 명확하고 일관적이다.

## [MODIFY] 5.3 profile 선택 로직 — "latest 파일 자동 선택"의 안전성

v0.39 §5는 5.3이 profile을 선택하는 우선순위를 다음과 같이 정의한다.

```text
1. 사용자가 지정한 profile 파일이 있으면 사용
2. 없으면 artifacts/child_page_discovery_profile_*_KST.json 중 최신 파일 사용
3. 없으면 5.0.1 수행 필요 안내
4. 사용자 요청 시 5.3 내부 fallback discovery 수행
```

- 현재 문안: "최신 파일"의 기준이 파일명 timestamp인지 파일 수정시각(mtime)인지 미정의.
- 수정 권고: "최신"의 판별 기준을 **파일명 끝 `_YYYYMMDD_HHMM_KST` 토큰 파싱** 기준으로 명시. mtime은 복사/이동 시 변동되므로 부적합.
- 추가 권고: 자동 선택된 profile 경로를 실행 로그에 **반드시 출력**하여, 사용자가 "어느 profile이 적용됐는지" 확인 가능하도록 한다. (silent 자동선택은 잘못된 주차 profile 적용을 눈치채지 못하게 만들 수 있음.)

수정 이유: 자동 선택 자체는 편의상 유지하되, "무엇이 선택됐는지 보이게" 만들어야 안전하다.
