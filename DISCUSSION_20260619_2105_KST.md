## 0. 논의 출발점

L1SW 파이프라인을 한 줄로 못 박으면:

```
.sdm → 모듈/시간 필터 → 80% 축소된 _l1sw.txt → 사람이 읽는 HTML 리포트.  여기서 끝.
```

이 "여기서 끝"의 지점들이 곧 5.5의 영토다.
선행 비교 분석은 공백을 4개(분류/저장/참조/taxonomy)로 정리했으나,
실제 RCA 작업 흐름 순서로 보면 더 잘게 7개로 쪼개진다.

## 1. L1SW 빈 부분 전수 — 7개 영역

각 영역을 "L1SW가 어디서 멈추는가 → 5.5가 무엇으로 메우는가" 형태로 정리.

| # | 영역 | L1SW가 멈추는 지점 | 5.5가 메우는 자산 |
|---|------|---------------------|---------------------|
| ① | 입력의 끝점 | 80% 줄여주지만 모듈 prefix(`[DSSM]`)로만 봄. "무슨 문제"인지는 사람 머릿속 | issue_type 자동 분류 (trigger signature, 예: `PHY_TIMER_EXPIRY`) |
| ② | 증거의 구조화 | HTML은 읽고 닫으면 끝. 기계가 다시 읽을 형태로 안 남음 | `log_patterns` (pattern_id / line_range / frequency) |
| ③ | 원인의 분류 체계 | severity 배지(색상)까지만. 표준 원인 라벨 없음 | `root_cause_categories` (10종) taxonomy |
| ④ | 과거와의 연결 | HTML 재참조 메커니즘 없음. 3개월 전 분석도 못 찾음 | case index + `similar_cases` 검색 |
| ⑤ | 외부 자산 연결 | Jira/HLD/TC/API 연결 필드 없음 | `related` 필드 (L1a 5.1·5.4와 연결되는 지점) |
| ⑥ | 재발 방지로의 전환 | 분석 1건 → 재사용 규칙/TC 후보로 승격되는 경로 없음 | `prevent_rule` + `skill_seed` 누적 |
| ⑦ | 신뢰도/검토 상태 | 추정/확정/리뷰 여부 추적 없음 | `confidence`(low~confirmed) + `review_status` |

→ 현재 `rca_kg/` 설계는 이 7개를 **전부** 어느 정도 건드리고 있다. schema도 그렇게 짜여 있음.

## 2. 보강 관점에서 진짜 봐야 할 두 가지

위 7개는 "L1SW에 없으니 5.5가 채운다"는 **정적 비교**다.
"더 보강"의 핵심은 여기 안 보이는 두 가지:

### (A) 7개 중 일부는 L1SW와 경계가 모호하다 — 특히 ⑥
L1SW도 "skill 보강 방식으로 지식 누적"을 한다고 확인됨(상세 구조는 6/20 확인 예정).
- L1SW가 **모듈 기반**으로 skill을 보강 → 5.5의 **issue_type 기반** skills_seed와 안 겹침 → 현 설계 유지
- L1SW가 **이슈 패턴까지** 누적 → ⑥ 중복 위험 → skills_seed 역할 재정의 필요

**→ ⑥은 6/20 확인 전까지 보강 설계를 확정하면 안 되는 영역.**

### (B) 7개가 다 똑같이 가치 있는 게 아니다 — 4개 층으로 분류

| 층 | 영역 | 성격 |
|----|------|------|
| L1SW가 구조적으로 못 하고, 효과 큰 것 | ③ taxonomy, ④ 과거 케이스 검색, ⑤ 외부 자산 연결 | L1SW 설계 철학(모듈 기반·HTML)상 불가능 → 5.5만의 순수 가치 |
| 출력만 받으면 비교적 쉬운 것 | ① 분류, ② 구조화 | `_l1sw.txt`만 들어오면 거의 기계적 |
| 경계 모호 / 확인 필요 | ⑥ skill 누적 | 위 (A) — 6/20 확인까지 보류 |
| 있으면 좋지만 부차적 | ⑦ 신뢰도/검토 | 팀 배포 단계에서 의미 커짐. 초기엔 과함 |

## 3. 보강 우선순위 제안 (미확정 — 6/20 확인 후 확정)

1. **1순위:** ④ 과거 케이스 검색 + ⑤ 외부 자산 연결
   - L1SW가 구조적으로 못 하고, L1a 다른 토픽(5.1 Jira 루프, 5.4 HLD)과 시너지 최대
2. **2순위:** ③ taxonomy 정교화
   - 분류 품질을 결정하는 기반
3. **검증만:** ①② — 이미 설계가 탄탄. 실 로그 end-to-end가 곧 검증
4. **보류:** ⑥ — 6/20 L1SW skill 보강 구조 확인까지
5. **연기:** ⑦ — 팀 배포 직전으로
