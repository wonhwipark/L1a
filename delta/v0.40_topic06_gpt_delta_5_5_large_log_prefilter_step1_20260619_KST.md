---

## 1. 배경

현재 RCA 분석 흐름은 다음과 같다.

```text
log txt 추출
  ↓
문제 원인 파악 요청
  ↓
기존 Skill 참조(현재 없음)
  ↓
Claude Code / Roo Code 분석
```

실제 로그 txt 크기는 작게는 100MB, 크게는 1GB 수준까지 커질 수 있다.
이 경우 원본 로그 전체를 Claude/Roo Context에 직접 전달하면 context 초과, 잡음 증가, 분석 비용 증가, 재사용 불가 문제가 발생한다.

또한 현재는 RCA 결과가 정형 지식으로 누적되지 않아, 동일한 `rach failure`, `scgFailure`, `tx abnormal`, `L2 max retransmission`, `crash` 유형을 매번 처음부터 분석하는 문제가 있다.

---

## 2. Step 1 목표

Step 1의 목표는 완성형 RCA Knowledge Graph 구현이 아니라, 즉시 적용 가능한 lightweight pre-filter와 RCA Markdown 저장 규칙을 정의하는 것이다.

```text
현재:
log.txt (100MB~1GB) → Claude/Roo 분석 요청 → 결과 휘발

개선 Step 1:
log.txt → issue-type pre-filter → signal_<issue_type>.txt → Claude/Roo 분석 → RCA_<date>_<issue_type>.md 저장
```

Step 1 산출물은 향후 RCA Skill Seed와 Knowledge Graph Seed로 사용한다.

---

## 3. 설계 원칙

1. 대용량 로그 원문 전체를 AI Context에 직접 입력하지 않는다.
2. 문제 유형별 keyword/signature 기반 pre-filter로 signal 파일을 먼저 만든다.
3. signal 파일에는 가능한 경우 원본 line number, timestamp, module/component, severity, UE/session/correlation id를 포함한다.
4. Claude/Roo에는 `log.txt`가 아니라 `signal_<issue_type>.txt`와 RCA 작성 템플릿을 전달한다.
5. RCA 결과는 정형 Markdown으로 저장하여 다음 분석에서 `@파일명` 참조 가능한 임시 Skill처럼 사용한다.
6. 이번 Step 1에서는 실제 Python streaming package, graph DB, embedding 검색, 자동 Skill 생성은 제외한다.

## 4. 문제 유형별 Pre-filter Rule 초안

### 4.1 rach failure

```text
RACH
rach_failure
msg1
msg2
msg3
msg4
RAR
preamble
```

우선 확인할 evidence:

- RACH attempt 시작 시점
- msg1~msg4 진행 단계
- RAR timeout 또는 preamble 실패
- 동일 UE/session/correlation id 기준 직전/직후 event

### 4.2 scgFailure

```text
SCG
scgFailure
PSCell
secondary
B1
B2
handover
```

우선 확인할 evidence:

- SCG add/reconfig 시작 시점
- PSCell 또는 secondary cell 관련 event
- B1/B2 measurement event
- handover/reconfiguration 전후
- failure 후 fallback 또는 release event

### 4.3 tx abnormal

```text
TX abnormal
tx fail
ul grant
harq
```

우선 확인할 evidence:

- TX abnormal 최초 발생 시점
- UL grant 전후
- HARQ fail/retry 흐름
- 동일 bearer/session/correlation id 기준 burst 구간

### 4.4 L2 max retransmission

```text
max retransmission
RLC
HARQ
retx
```

우선 확인할 evidence:

- max retransmission 발생 시점
- RLC/HARQ retransmission burst
- 동일 UE/RB/session 기준 반복 횟수
- radio link 품질 또는 scheduler 관련 직전 event

### 4.5 crash

```text
ASSERT
FATAL
crash
backtrace
PC=
LR=
stack
```

우선 확인할 evidence:

- crash/assert/fatal 최초 line
- call stack/backtrace 전체
- PC/LR 또는 symbol 정보
- crash 직전 동일 module의 error burst
- crash 전후 N라인 evidence window

## 9. 500회 이상 반복 분석을 위한 파일 기반 Knowledge Graph 운영안

앞으로 동일 계열 로그를 500회 이상 분석한다고 가정하면, Step 1의 pre-filter 결과와 RCA Markdown은 단발 산출물이 아니라 장기 Knowledge Graph의 seed로 운영해야 한다.

초기부터 Neo4j 같은 graph DB를 도입하기보다, 이 프로젝트에서는 LLM이 직접 읽고 Git으로 변경 이력을 관리할 수 있는 구조화 텍스트 기반 Knowledge Graph가 가장 적합하다.

```text
rca_kg/
  ├─ cases/
  │  ├─ 2026-06-19_rach_failure_001.yaml
  │  ├─ 2026-06-20_scg_failure_002.yaml
  │  └─ ...
  ├─ signals/
  │  ├─ 2026-06-19_rach_failure_001_signal.txt
  │  └─ ...
  ├─ skills_seed/
  │  ├─ rach_failure_analyzer.md
  │  ├─ scg_failure_analyzer.md
  │  ├─ tx_abnormal_analyzer.md
  │  ├─ l2_retx_analyzer.md
  │  └─ crash_analyzer.md
  ├─ indexes/
  │  ├─ index.md
  │  ├─ by_issue_type.yaml
  │  ├─ by_root_cause.yaml
  │  ├─ by_log_pattern.yaml
  │  ├─ by_api.yaml
  │  └─ by_fix.yaml
  └─ schema/
     ├─ rca_case.schema.yaml
     └─ taxonomy.yaml
```

### 9.1 저장 원칙

1. 원본 대용량 `log.txt`는 Knowledge Graph에 직접 저장하지 않는다.
2. 원본 로그는 별도 raw log 위치에 보관하고, Knowledge Graph에는 path, size, checksum, time range만 저장한다.
3. `signal_<issue_type>.txt`는 pre-filter 결과로 보관한다.
4. 분석 1건은 `cases/<case_id>.yaml` 1개로 저장한다.
5. 사람이 읽고 리뷰하기 쉬운 RCA Markdown은 Skill Seed 또는 보고용 산출물로 함께 유지한다.
6. 500건 이상 검색을 위해 전체 case YAML을 매번 읽지 않고 `indexes/*.yaml`을 먼저 조회한다.

### 9.2 Case YAML 예시

```yaml
case_id: 2026-06-19_rach_failure_001
issue_type: rach_failure
status: analyzed

source:
  original_log:
    path: logs/raw/2026-06-19_rach_failure_001.txt
    size_bytes: 823000000
    checksum: sha256:...
  signal_file: signals/2026-06-19_rach_failure_001_signal.txt

symptom:
  occurred_at: "2026-06-19T14:22:03+09:00"
  module:
    - L1
    - RRC
  severity: ERROR
  ue_id: masked_ue_001
  session_id: masked_session_001
  correlation_id: masked_corr_001

log_patterns:
  - pattern_id: lp_rach_msg2_timeout
    normalized: "RACH attempt failed: msg2 timeout"
    raw_examples:
      - "[14:22:02] RACH attempt #4 failed, reason=msg2 timeout"
    line_range: "120340-120520"
    frequency: 4

root_cause:
  category: radio_access_timeout
  summary: "RAR/msg2 응답 지연으로 RACH procedure 실패"
  confidence: medium
  evidence:
    - lp_rach_msg2_timeout

related:
  jira: []
  hld: []
  tc: []
  api:
    - xxx_RACH_REQ
    - xxx_RACH_CNF

fix:
  summary: "RACH timeout 관련 파라미터 및 retry 조건 점검 필요"
  commit: null

prevent_rule:
  - rule: "msg2 timeout 3회 이상 연속 발생 시 RACH parameter 확인"
    tc_candidate: "RACH msg2 timeout retry scenario 추가"

skill_seed:
  update_target: skills_seed/rach_failure_analyzer.md
  checklist_candidate:
    - "msg1~msg4 단계 중 최초 실패 지점 확인"
    - "RAR timeout 반복 횟수 확인"
    - "동일 UE/session/correlation id timeline 확인"

review:
  status: draft
  reviewer: ""
  reviewed_at: ""

links:
  similar_cases: []
  supersedes: []
  duplicates: []
```

### 9.3 Index 운영안

500건 이상이 되면 전체 YAML을 매번 LLM에 넣는 방식은 비효율적이다. 먼저 index를 조회해 유사 case 후보 3~5개만 골라 Claude/Roo Context에 넣는다.

`indexes/by_issue_type.yaml` 예시:

```yaml
rach_failure:
  - case_id: 2026-06-19_rach_failure_001
    root_cause: radio_access_timeout
    pattern: lp_rach_msg2_timeout
    confidence: medium
scg_failure:
  - case_id: 2026-06-20_scg_failure_002
    root_cause: nr_reconfig_timeout
    pattern: lp_scg_failure_after_reconfig
```

`indexes/by_log_pattern.yaml` 예시:

```yaml
lp_rach_msg2_timeout:
  normalized: "RACH attempt failed: msg2 timeout"
  cases:
    - 2026-06-19_rach_failure_001
    - 2026-07-02_rach_failure_014
```

`indexes/by_root_cause.yaml` 예시:

```yaml
radio_access_timeout:
  cases:
    - 2026-06-19_rach_failure_001
  prevent_rules:
    - "msg2 timeout 3회 이상 연속 발생 시 RACH parameter 확인"
```

### 9.4 500회 반복 분석 운영 흐름

```text
1. log.txt 수집
2. issue type 지정
3. pre-filter 실행
4. signal_<issue_type>.txt 생성
5. indexes/*.yaml에서 유사 case 후보 3~5개 검색
6. Claude/Roo 입력 구성
   - signal file
   - 유사 case YAML 일부
   - issue type별 skills_seed Markdown
7. RCA 분석
8. case YAML 저장
9. index 업데이트
10. skills_seed 업데이트 후보 생성
```

### 9.5 Skill Seed 운영안

문제 유형별로 `skills_seed/*.md`를 두고 반복 분석 checklist와 root cause category를 누적한다.

예: `skills_seed/rach_failure_analyzer.md`

```markdown
# rach_failure_analyzer

## Trigger Signatures
- RACH
- msg2 timeout
- RAR timeout
- preamble failure

## Required Evidence
- msg1~msg4 progression
- RAR response 여부
- UE/session/correlation id timeline
- 동일 cell에서 반복 발생 여부

## Root Cause Categories
- radio_access_timeout
- preamble_collision
- parameter_mismatch
- cell_load_issue
- scheduling_delay

## Analysis Checklist
1. msg1~msg4 중 최초 실패 단계 확인
2. 동일 UE 기준 재시도 횟수 확인
3. RAR timeout 반복 여부 확인
4. cell 단위 다발성인지 단일 UE 이슈인지 구분
5. 관련 API/IPC 후보 확인

## Prevent Rule Candidates
- msg2 timeout 3회 이상 연속 발생 시 RACH parameter 점검
- 동일 cell 다수 UE에서 RACH failure 발생 시 cell load 확인
```

초기에는 RCA Markdown과 case YAML을 사람이 검토하고, 10~20건 이상 쌓이면 공통 trigger, evidence, checklist를 수동/반자동으로 `skills_seed/*.md`에 반영한다.

### 9.6 안정성 기준

```text
  - LLM 입력은 index + 후보 case + signal 중심으로 제한
```

### 9.8 결론

5.5의 현실적 Knowledge Graph는 초기부터 graph DB를 구축하는 방식이 아니라, 분석 1건이 끝날 때마다 다음 분석을 더 빠르게 만드는 파일 기반 지식 축적 시스템으로 시작하는 것이 좋다.

```text
case YAML + signal txt + index YAML/Markdown + skill seed Markdown
```

이 구조는 Claude/Roo가 직접 읽을 수 있고, Git으로 변경 이력을 관리할 수 있으며, 500건 이상으로 증가해도 SQLite나 graph DB로 확장 가능한 source of truth 역할을 한다.
