# 2. 5.1 Jira Feedback Loop 변경안

## 2.1 기존 흐름

```text
Critical Defect
↓
Jira
↓
Review
↓
Jira Close
↓
Root Cause
↓
Prevent Rule
↓
Skill Update
```

## 2.2 변경 흐름

```text
(1) Critical Defect Detection
    ↓
(2) AI Root Cause Analysis
    ↓
(3) AI Patch Proposal
    ↓
(4) P4 Shelve Creation
    ↓
(5) Jira Creation
    - Shelve Number 포함
    - Root Cause Summary 포함
    ↓
(6) Review & Discussion
    ↓
(7) Resolution Decision
    ├─ Reject
    │   ↓
    │   False Positive Rule 정리
    │   ↓
    │   Skill Update
    │
    └─ Resolve
        ↓
        Prevent Rule 정리
        ↓
        Skill Update
```

# 3. 단계별 의미

| Step | 이름 | 설명 |
|------|------|------|
| 1 | Critical Defect Detection | 코드 변경 영역에서 Critical defect 후보를 검출한다. |
| 2 | AI Root Cause Analysis | AI가 결함 후보의 원인과 영향 범위를 분석한다. |
| 3 | AI Patch Proposal | AI가 수정 방향과 코드 변경안을 제안한다. |
| 4 | P4 Shelve Creation | 제안 수정사항을 바탕으로 P4 shelve를 생성한다. |
| 5 | Jira Creation | Jira 등록 시 shelve number와 root cause summary를 명시한다. |
| 6 | Review & Discussion | Reviewer가 결함 여부와 수정 방향을 검토한다. |
| 7 | Resolution Decision | 실제 수정으로 이어지지 않으면 Reject, 수정 반영이면 Resolve로 분기한다. |

# 4. P4 Shelve 반영 원칙

5.1에서는 `AI Patch Proposal` 이후 `P4 Shelve Creation`을 명시 단계로 둔다.

Jira 등록 시 아래 정보가 반드시 포함되어야 한다.

```text
- P4 Shelve Number
- AI Root Cause Summary
- AI Patch Proposal Summary
- Review 대상 파일/함수
```

이유는 Jira만으로는 실제 제안 수정사항과 코드 변경 후보를 추적하기 어렵기 때문이다.
P4 shelve number를 Jira에 명시해야 Reviewer가 바로 수정 후보를 확인할 수 있다.

# 5. Review 결과 분기

## 5.1 Reject

Reject는 실제 수정으로 이어지지 않은 경우이다.

```text
Review & Discussion
↓
Resolution Decision = Reject
↓
False Positive Rule 정리
↓
Skill Update
```

Reject 사례는 다음 defect 검출 시 과탐을 줄이기 위한 기준으로 사용한다.

예시:

```text
- 상위 layer에서 serialization이 보장되는 경우 mutex 누락으로 판단하지 않는다.
- 특정 domain에서 invalid value는 초기화 sentinel로 사용되므로 defect로 판단하지 않는다.
```

## 5.2 Resolve

Resolve는 실제 수정이 반영된 경우이다.

```text
Review & Discussion
↓
Resolution Decision = Resolve
↓
Prevent Rule 정리
↓
Skill Update
```

Resolve 사례는 다음 defect 검출 시 실제 결함 패턴으로 재사용한다.

예시:

```text
- Peer RAT/domain이 존재하는 경우 AS/PS conflict를 RF API 호출 전에 확인한다.
- ENDC MCG/SCG 우선순위 조건을 switch decision 이전에 검증한다.
```

# 7. Skill과 Knowledge의 역할 분리

장기적으로는 Skill과 Knowledge DB를 분리한다.

```text
Skill
= Thinking Rule

Knowledge DB
= Experience
```

## 7.1 Skill

Skill은 AI가 defect를 판단할 때 따라야 하는 사고 방식과 검토 규칙을 정의한다.

```text
예:
Peer RAT 존재 시
- AS conflict 확인
- PS conflict 확인
- MCG/SCG priority 확인
- RF API 호출 전 conflict resolver 통과 여부 확인
```

## 7.2 Knowledge DB

Knowledge DB는 실제 Resolve/Reject 경험을 저장한다.

```text
예:
- Resolved defect case
- Rejected false positive case
- Jira link
- P4 shelve number
- Review comment
- Prevent rule
- False positive rule
```

# 10. Index-based Knowledge System 방향

년 단위로 사례가 누적될 경우 단순 Markdown 누적만으로는 검색과 AI context 주입에 한계가 있다.

따라서 원칙은 아래와 같다.

```text
저장은 많이 한다.
하지만 AI에게는 필요한 것만 검색해서 넣는다.
```

Defect 검출 시 전체 Knowledge DB를 읽지 않고 아래 순서로 사용한다.

```text
1. 변경 파일 / 함수 / 도메인 / defect type 추출
2. Metadata index 검색
3. 유사 Resolve case Top K 조회
4. 유사 Reject case Top K 조회
5. 관련 Prevent Rule 조회
6. 선별된 내용만 AI context에 주입
```
