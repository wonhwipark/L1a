## 1. Background

기존 `5.3 Weekly Report Collection` 프롬프트는 Confluence Parent Page 아래의 Child Page를 수집한다는 전제를 가지고 있다.

하지만 실제 운영에서는 다음 문제가 있다.

```text
- 주간보고 Parent Page는 매주 새로 생성된다.
- Parent Page link는 매주 사용자가 직접 제공한다.
- Child Page 목록도 매주 달라질 수 있다.
- 따라서 특정 Parent Page ID나 특정 Child Page URL 목록을 저장해도 다음 주에 재사용 가치가 낮다.
- 반면 Child Page를 찾는 방식은 재사용 가치가 높다.
```

따라서 Child Page Discovery를 5.3 내부에만 두지 않고, `5.0.1` 공통 프롬프트로 분리하여 재사용 가능한 Discovery Strategy를 먼저 검증하는 구조로 정리한다.

## 2. Key Decision

### 2.1 `5.0.1` 신규 추가

신규 프롬프트를 추가한다.

```text
automation_prompt/5_0_1_confluence_child_page_discovery_strategy_prompt.md
```

역할은 다음과 같다.

```text
Parent Page link를 입력받아 Confluence Child Page를 찾는 여러 방법을 순차적으로 시도하고,
성공한 Child Page Discovery 방식을 strategy profile로 저장한다.
```

---

### 2.2 `5.0.1`은 특정 주차 데이터를 저장하지 않는다

`5.0.1` 산출물은 아래 정보를 저장하지 않는다.

```text
- 특정 주차 Parent Page ID
- 특정 주차 Parent Page URL
- 특정 주차 Child Page URL 목록
- 특정 주차 Child Page ID 목록
```

이유는 주간보고 Parent Page가 매주 새로 생성되기 때문이다.

---

### 2.3 `5.0.1`은 Child Page를 찾는 방식만 저장한다

저장 대상은 다음과 같다.

```text
- last_success_method
- confidence
- strategy input type
- discovery query type
- child filter rule
- fallback method list
```

예상 profile 구조:

```json
{
  "profile_type": "confluence_child_page_discovery_strategy",
  "created_at_kst": "2026-06-17 01:10 KST",
  "last_success_method": "confluence_rest_descendants",
  "confidence": "high",
  "strategy": {
    "input_type": "parent_page_url",
    "resolve_parent_page_id_at_runtime": true,
    "discovery_query_type": "descendants",
    "child_filter": {
      "exclude_parent": true,
      "include_descendants": true
    }
  },
  "fallback_methods": [
    "confluence_mcp_child_pages",
    "parent_body_link_extraction",
    "label_search",
    "title_pattern_search",
    "fuzzy_search",
    "user_assisted_mode"
  ]
}
```

## 3. Output Artifact Rule

모든 output artifact 파일명 끝에는 저장 시점의 KST 기준 날짜와 시간을 반드시 추가한다.

형식:

```text
<base_filename>_<YYYYMMDD>_<HHMM>_KST.<ext>
```

따라서 `5.0.1`의 필수 산출물은 다음과 같다.

```text
artifacts/child_page_discovery_profile_<YYYYMMDD>_<HHMM>_KST.json
```

고정 파일명인 아래 형태는 기본 artifact로 사용하지 않는다.

```text
artifacts/child_page_discovery_profile.json
```

대신 `5.3`은 실행 시 `artifacts/child_page_discovery_profile_*_KST.json` 패턴에서 최신 profile을 선택하거나, 사용자가 profile 파일 경로를 직접 지정할 수 있도록 한다.

## 4. `5.0.1` Prompt Scope

### 4.1 Input

```text
- Parent Page Confluence Link
- Confluence access method: MCP + REST API
- dry-run
```

선택 입력:

```text
- Current Week
```

`Previous Week Page Link`는 5.0.1의 필수 입력이 아니다.  
이전 주 양식 분석은 5.3의 역할이다.

---

### 4.2 Discovery Method Priority

`5.0.1`은 아래 방법을 순차적으로 시도한다.

```text
1. Confluence REST API parent / ancestor / descendants / children
2. Confluence MCP child page 조회
3. Parent Page 본문 링크 추출
4. Label Search
5. Title Pattern Search
6. Fuzzy Search
7. User Assisted Mode
```

한 방법이 실패해도 종료하지 않고 다음 방법을 시도한다.

성공 시 다음 정보를 기록한다.

```text
- 성공 method
- confidence
- runtime에 필요한 입력 형태
- fallback method list
```

## 5. `5.3 Weekly Report Collection` 변경 방향

`5.3`은 Child Page Discovery를 처음부터 매번 수행하지 않는다.

우선순위는 다음과 같다.

```text
1. 사용자가 지정한 discovery profile 파일이 있으면 해당 profile 사용
2. 지정이 없으면 artifacts/child_page_discovery_profile_*_KST.json 중 최신 파일 사용
3. profile이 없으면 5.0.1 수행이 필요하다고 안내
4. 단, 사용자가 요청하면 fallback discovery를 5.3 내부에서도 수행 가능
```

---

## 6. `5.3` Runtime Input

매주 사용자가 제공하는 입력은 다음과 같다.

```text
- Current Week
- 이번 주 Parent Page Confluence Link
- Previous Week Page Confluence Link
- dry-run
```

팀원 목록은 사용하지 않는다.

```text
- members.yaml 없음
- missing member detection 없음
- 실제 발견된 Child Page 내용만 사용
```

## 11. Rationale

이 구조는 다음 원칙에 부합한다.

```text
- 5.0은 공통 실행 프레임워크

- 모든 artifact는 KST timestamp를 포함
- 최종 목표는 Python 자동화 도구이며, 현재는 구현 전 프롬프트 정리 단계
```
