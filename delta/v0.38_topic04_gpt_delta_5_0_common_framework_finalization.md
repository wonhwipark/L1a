## 6. Adaptive Schema Decision

Adaptive Schema should prioritize open-source, proven, stable and efficient options.

Selection Priority:

```text
1. Open-source and widely adopted
2. Proven stability in real projects
3. Efficient for long-term operation
4. Easy to maintain and inspect
```

The previous candidate item `Compatible with current environment` was intentionally removed from the selection priority because environment compatibility is already handled by Environment Discovery and Capability Discovery.

Candidate Storage Schema:

```text
Tier 1: SQLite, Markdown + Index
Tier 2: YAML, JSON
```

Validation Layer:

```text
Pydantic
Dataclass
```

Pydantic and Dataclass are validation/structure layers, not storage schema candidates.

---

## 7. Artifact Management Decision

Artifacts must be stored under timestamp-based run folders to avoid overwrite and preserve execution history.

Timestamp format:

```text
YYYYMMDD_HHMM_KST
```

Directory rule:

```text
artifacts/
run_<timestamp>/
```

Naming rule:

```text
<artifact_name>_<timestamp>.<ext>
```

Example:

```text
artifacts/run_20260616_0215_KST/analysis_report_20260616_0215_KST.md
```

---

## 8. Layered Architecture Decision

Separate policy, execution, artifacts and knowledge.

```text
Skill Layer
        ↓
Workflow Layer
        ↓
Artifact Layer
        ↓
Knowledge Layer
```

Ownership:

```text
Skill Layer      = Policy Owner
Workflow Layer   = Policy Consumer
Artifact Layer   = Execution History
Knowledge Layer  = Long-term Memory
```

Important rule:

```text
Workflow modules are policy consumers, not policy owners.
```

---

## 9. Skill Loading Strategy Decision

Workflow modules must not hardcode team policy.  
Policies are dynamically loaded from skills.

Search order:

```text
Branch Skill
    ↓
Repository Skill
    ↓
Common Skill
    ↓
Default Skill
```

Recommended skill format:

```text
Tier 1: Markdown
Tier 2: YAML
Tier 3: JSON
```

Team-defined skills and policies are authoritative.
